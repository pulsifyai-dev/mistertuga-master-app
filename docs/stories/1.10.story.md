# Story 1.10: Go-Live + 48h Monitoring + Rollback Readiness

**Epic:** EPIC-1 (Firebase → Supabase Quality Migration)
**Sprint:** 3 (Day 21+)
**Effort:** 0.5 day + 48h monitoring
**Priority:** P0
**Status:** InProgress
**Agent:** @devops (Gage)

---

## Story

**As a** project owner,
**I want** the migration to go live with active monitoring and a proven rollback plan,
**So that** any issues are caught quickly and can be resolved without data loss.

---

## Context

With the application deployed (Story 1.9) and QA verified (Story 1.8), this story manages the go-live process: DNS cutover, user communication, 48-hour monitoring period, and eventual Firebase decommission after 2 weeks. The rollback plan keeps Firebase live as a warm standby.

**Reference:** `docs/architecture/technical-debt-assessment.md` §6 Post-Migration Monitoring, §7 Rollback Strategy

---

## Acceptance Criteria

- [x] **AC1:** DNS/routing switched to new platform
  - Production URL: `mistertuga.vercel.app` — verified live, serving login page
  - HTTPS verified (Vercel auto-SSL)
  - Vercel auto-deploys from `main` branch (GitHub integration)
  - Old Firebase Hosting URL still accessible (rollback)

- [ ] **AC2:** Password reset emails sent to all users
  - All existing users receive password reset email
  - Email includes clear instructions and timeline
  - Support contact provided for issues

- [ ] **AC3:** 48h monitoring active using Phase 5 checklist
  - RLS policy denials: alert if > 10/hour
  - Realtime connection count: alert if > 150 (of 200 limit)
  - Query latency (p95): alert if > 500ms
  - Failed webhooks: alert on any failure
  - Auth failures: alert if > 20/hour
  - Database size: alert if > 400MB (of 500MB free tier)

- [ ] **AC4:** Monitoring dashboard or alerts configured (TD-H10)
  - Supabase Dashboard bookmarked with relevant views
  - Alert notifications configured (email or Slack)
  - Runbook for each alert type documented

- [ ] **AC5:** Caching strategy implemented (TD-M8)
  - Supabase query results cached where appropriate
  - Static assets cached via hosting platform CDN
  - Cache invalidation strategy documented

- [ ] **AC6:** Rollback plan tested and documented
  - Rollback procedure: revert DNS → Firebase serves traffic
  - Estimated rollback time: < 5 minutes
  - Data sync procedure if orders created during Supabase period
  - Rollback tested in staging before go-live

- [ ] **AC7:** Firebase decommission scheduled
  - Calendar event: 2 weeks post go-live
  - Firebase decommission checklist:
    - Verify no traffic to Firebase Hosting
    - Export final Firestore backup
    - Disable Firebase Auth
    - Delete Cloud Functions
    - Document Firebase project for archival

## Rollback Decision Matrix

| Scenario | Action | Timeline |
|----------|--------|----------|
| Minor UI bugs | Hotfix on Supabase app | Same day |
| Auth issues (users can't login) | Extend password reset window | Same day |
| Data integrity issue | Investigate, rollback if critical | 2-4 hours |
| RLS blocking legitimate operations | Fix policies, no rollback needed | 1-2 hours |
| Supabase outage | Rollback to Firebase | < 5 minutes |
| Critical data loss | Rollback to Firebase immediately | < 5 minutes |
