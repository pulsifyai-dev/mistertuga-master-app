## Story

**As a** developer,
**I want** all Firebase data hooks rewritten to use Supabase client with Realtime subscriptions,
**So that** the application reads from and writes to the new PostgreSQL database.

---

## Context

After Story 1.4 decomposes the orders page, the Firebase hooks are isolated in 4 hook files + Server Actions. This story rewrites them to Supabase:
- `useCollection()` → Supabase Realtime channels
- `useDoc()` → Supabase `.select().single()` + Realtime
- Server Actions: Firebase Admin SDK → Supabase service client

The decomposition from Story 1.4 makes this rewrite manageable — each hook is ~80-120 lines instead of embedded in 1,704 lines.

**Reference:** `docs/architecture/technical-debt-assessment.md` §3.3 Phase C

## Dev Notes

- **Data mapping:** Supabase normalized schema (orders → customers → order_items) mapped back to frontend `Order` type with embedded customer + items
- **Status mapping:** Supabase `status: "open"` → UI `"Pending Production"`, `status: "fulfilled"` → UI `"Shipped"`
- **Realtime strategy:** Full re-fetch on any `postgres_changes` event (simpler and more reliable than row-level patching for joined data)
- **Bundle size improvement:** Orders page 562→447 kB (-115 kB), Settings 272→157 kB (-115 kB) by removing Firebase client SDK
- **TypeScript:** Clean compilation (0 errors)
- **Next.js build:** All routes compile successfully
- **Profit stats deferred:** Uses custom dummy data in Firebase, not a real data layer. Proper implementation in EPIC-2 Story 2.7.
