## Story

**As an** admin user,
**I want** daily revenue automatically computed from stored orders and displayed in charts and tables,
**So that** I can track revenue trends without relying on the n8n "Faturamento" workflow.

---

## Context

The n8n "Faturamento e Taxas (Diario)" workflow currently fetches Shopify data and calculates daily revenue/taxes. This story re-implements that logic in-app, using the orders table as source of truth instead of hitting the Shopify API directly. The n8n workflow references: processFeesPercent, cogsPercent, fulfillmentPercent as cost parameters.

**Reference:** `docs/product/mistertuga_master.md` §6.2, n8n workflow "Faturamento e Taxas (Diario)"

  - Profit = Revenue - (Ad Spend + Manual Expenses + Fixed Monthly Costs)
  - Margin percentage displayed

- [x] **AC6:** Country breakdown view
  - Revenue split by country with progress bars and percentages
  - Order count per country
  - Sorted by revenue descending

## Dev Notes

- **Full rewrite of profit-stats page.** All dummy data (dummyProfitDoc, buildDummyDailyNetProfit, hardcoded expenses) removed. "UNDER CONSTRUCTION" banner removed.
- **Revenue source:** Aggregates from `orders` table using `shopify_created_at` for date grouping. Excludes soft-deleted orders.
- **Item count** calculated by joining with `order_items` table and summing quantities.
- **Unified profit formula:** `getProfitSummary()` queries 4 tables in one call: daily_revenue, ad_spend, manual_expenses, expense_categories.
- **AdSpendSummaryCard and ExpenseSummaryCard** files retained but no longer imported in profit page — their data is now folded into the unified dashboard.
- **Backfilling:** Click "Recalculate" with a date range to backfill historical revenue data from existing 576 orders.
- **Country breakdown** uses null `country_code` for aggregate rows and specific codes (PT, ES, DE) for per-country data.

**Must fix before Done:**
1. **CRITICAL:** `UNIQUE(date, country_code)` does NOT enforce uniqueness for NULL `country_code` in PostgreSQL — aggregate rows can be duplicated. Fix: use `UNIQUE NULLS NOT DISTINCT (date, country_code)` (Postgres 15+) or add partial unique index.
2. **CRITICAL:** Off-by-one date boundary in `revenue-engine.ts:39-40` — `.lt('23:59:59.999Z')` excludes sub-millisecond orders. Fix: use `.lt(nextDay + 'T00:00:00.000Z')`.

**Should fix:**
3. Missing index on `orders.shopify_created_at` — used for all revenue queries but not indexed
4. N+1 sequential queries in `calculateDateRange` — 90-day backfill = ~720 DB queries (timeout risk)
5. Cron route swallows countries query error silently (line 19)
6. No error state on dashboard when fetch fails — shows zeros with no explanation
7. Chart X-axis shows day-of-month only — confusing for 90-day view

**Noted (acceptable):**
8. Fixed costs not prorated by date range — same as Story 2.6 decision
9. Country filter on profit doesn't filter ad_spend (no country_code in ad_spend schema)
10. AdSpendSummaryCard and ExpenseSummaryCard are dead code — folded into unified dashboard
