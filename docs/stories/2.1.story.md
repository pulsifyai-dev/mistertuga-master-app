## Story

**As an** operations manager,
**I want** exchange/return requests from customer emails automatically captured in the app,
**So that** I can track and manage all exchanges from a single interface instead of handling emails manually.

---

## Context

The n8n AI agent already monitors incoming emails, detects exchange intent, and extracts structured data. Currently this data has no destination. This story creates the database schema for exchanges and exposes a webhook endpoint that n8n can call to write records into Supabase.

**Reference:** `docs/product/mistertuga_master.md` §4, `docs/architecture/schema-blueprint.md` (Phase 2: exchanges)
**n8n workflow:** `docs/product/` (reference only — contains secrets, do not commit)

---

## Acceptance Criteria

- [x] **AC1:** Exchanges table created via migration
  - Schema matches `docs/architecture/schema-blueprint.md`
  - All columns, constraints, indexes in place
  - Status enum: new, in_review, waiting_customer, approved, rejected, completed
  - Soft reference to orders table (order_number lookup, not strict FK — exchange may arrive before order sync)

- [x] **AC2:** Exchange attachments table created
  - Supports file uploads via Supabase Storage
  - Linked to exchange record
  - File type/size metadata stored

- [x] **AC3:** Webhook endpoint for n8n integration
  - API route: `POST /api/webhooks/exchanges`
  - Accepts JSON payload with: order_number, customer_name, customer_email, customer_phone, customer_address, reason, received_description, original_email_text
  - Validates required fields
  - Returns 201 on success, 400 on validation error
  - Idempotent: duplicate detection by order_number + created_at window (10min)

- [x] **AC4:** Webhook authentication
  - API key or HMAC-based authentication
  - Reject unauthenticated requests with 401
  - Rate limited: max 100 requests/hour

- [x] **AC5:** RLS policies on exchanges tables
  - Admin: full CRUD
  - Fornecedor: no access (exchanges are admin-only operations)
  - Service role: full (for webhook writes)

- [x] **AC6:** Exchange record links to existing order (when found)
  - On insert, attempt to match order_number to orders table
  - If found: set order_id FK
  - If not found: leave order_id NULL, log warning

- [x] **AC7:** Supabase Realtime enabled on exchanges
  - New exchange records appear in real-time in the UI (Story 2.2)

## Dev Notes

- **Migrations** use sequential timestamps `20260303150001-4` to ensure correct ordering
- **RLS** follows existing pattern from `20260302150004_create_rls_policies.sql` — admin-only access
- **Realtime** enabled in the exchanges table creation migration (same migration, like orders pattern)
- **Storage bucket** `exchange-attachments` is private with admin-only policies
- **Webhook** uses service role client to bypass RLS for inserts
- **Rate limit** reuses existing `src/lib/rate-limit.ts` (100 req/hour per IP)
- **Duplicate detection** uses 10-minute window on order_number to prevent n8n double-fires
- **Order linking** is soft — if order_number doesn't match, order_id stays NULL
- **Build verified** — `npx tsc --noEmit` and `npm run build` pass clean
