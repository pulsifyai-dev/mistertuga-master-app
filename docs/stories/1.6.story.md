## Story

**As a** user with assistive technology,
**I want** the application to be accessible with proper ARIA attributes, empty states, and loading feedback,
**So that** I can use MisterTuga Insights effectively regardless of how I interact with the interface.

---

## Context

The Brownfield Discovery found **zero ARIA attributes** across the entire application (TD-C9, upgraded to CRITICAL in Phase 6). Combined with missing empty states (TD-C10), no skip navigation (TD-H11), inaccessible charts (TD-H12), and unannounced toasts (TD-H13), the app has 0% WCAG compliance. This story brings the app to WCAG Level AA.

**Reference:** `docs/architecture/ux-specialist-review.md` §3-6, `docs/architecture/technical-debt-assessment.md` §2 (CRITICAL items)

## Acceptance Criteria

- [x] **AC1:** Skip navigation link added (TD-H11)
  - Visually hidden link at top of page: "Skip to main content"
  - Visible on keyboard focus
  - Targets `<main>` element with `id="main-content"`
  - Test: Tab into page → first focusable element is skip link

- [x] **AC2:** All icon-only buttons have aria-label (TD-C9)
  - Audit all buttons without visible text (~30+ instances)
  - Add descriptive `aria-label` to each
  - Examples: "Edit order", "Delete order", "Export PDF", "Close dialog"
  - Test: Screen reader announces button purpose

- [x] **AC3:** Form validation uses aria-describedby (TD-C9)
  - Error messages linked to inputs via `aria-describedby`
  - `aria-invalid="true"` on fields with errors
  - Test: Screen reader announces error when field is focused
  - NOTE: Already implemented by shadcn FormControl component

- [x] **AC4:** Toaster component uses aria-live region (TD-H13)
  - `aria-live="polite"` on toast container
  - `role="status"` for success toasts
  - `role="alert"` for error toasts
  - Test: Screen reader announces toast messages

- [x] **AC5:** Charts have accessible descriptions (TD-H12)
  - `role="img"` on chart containers
  - `aria-label` with data summary (e.g., "Profit chart: 1,234 EUR total this month")
  - Hidden data table alternative for screen readers
  - Test: Screen reader conveys chart meaning

- [x] **AC6:** Active navigation has aria-current (TD-C9)
  - `aria-current="page"` on active sidebar link
  - Test: Screen reader announces which page is active

- [x] **AC7:** Empty states for all data views (TD-C10)
  - Orders list: "No orders found" with illustration
  - Search results: "No orders match your search"
  - Filtered views: "No orders in [country]"
  - Charts: "No data for this period"
  - Each empty state suggests an action

- [x] **AC8:** Error boundary components added (TD-H7)
  - Root error boundary catches unhandled errors
  - Per-section error boundaries for orders, charts, settings
  - User-friendly error messages with retry action

- [x] **AC9:** Skeleton loaders replace spinners (TD-M15)
  - Orders table: row skeletons matching table layout
  - Charts: skeleton matching chart dimensions
  - Settings: form field skeletons
  - Test: Loading state shows content-shaped skeletons, not spinner

- [x] **AC10:** Mobile-optimized table view (TD-M16)
  - Orders table readable on mobile (card layout or horizontal scroll with sticky columns)
  - Critical fields visible without scrolling: order ID, customer, status
