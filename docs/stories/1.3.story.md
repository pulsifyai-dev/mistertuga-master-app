## Story

**As a** project owner,
**I want** all existing Firestore data migrated to Supabase PostgreSQL with verified integrity,
**So that** no order, customer, or configuration data is lost during the platform transition.

---

## Context

Firestore uses a subcollection pattern: `orders/{countryCode}/orders/{orderId}` with embedded customer data and items arrays. This must be flattened into normalized PostgreSQL tables. Customer deduplication by phone is required (Phase 5). ALL country subcollections must be explicitly exported — Firestore export can miss subcollections (Phase 5 risk).

**Reference:** `docs/architecture/SCHEMA.md` §4-6, `docs/architecture/db-specialist-review.md` §3

## Dev Notes

### Data Volume
- 576 orders across 3 countries (PT: 219, ES: 349, DE: 8)
- 1,961 order line items
- 563 unique customers (dedup by phone)
- Average 3.4 items per order

### Phone Normalization Strategy
- Numeric phones (535): converted to string, prepended with country prefix (+351, +34, +49)
- String phones (41): normalized, ensured + prefix
- All 576 orders had phone numbers — no missing data

### What Was NOT Migrated
1. **Firebase users** — IDs incompatible, admin already exists in Supabase
2. **Profit stats** — Aggregate snapshot data (Dec 2025), will be recalculated
3. **Expenses** — No individual expense records exist in Firestore
4. **Products** — No products collection; product data preserved in order_items

### Idempotency
- Export script can be re-run safely (overwrites JSON files)
- Load script uses ON CONFLICT DO NOTHING — safe to re-run (won't duplicate)
- Full pipeline is transactional (ROLLBACK on any error)
