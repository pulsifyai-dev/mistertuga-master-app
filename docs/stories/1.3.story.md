## Story

**As a** project owner,
**I want** all existing Firestore data migrated to Supabase PostgreSQL with verified integrity,
**So that** no order, customer, or configuration data is lost during the platform transition.

---

## Context

Firestore uses a subcollection pattern: `countries/{countryCode}/orders/{orderId}` with embedded customer data and items arrays. This must be flattened into normalized PostgreSQL tables. Customer deduplication by phone is required (Phase 5). ALL country subcollections must be explicitly exported — Firestore export can miss subcollections (Phase 5 risk).

**Reference:** `docs/architecture/SCHEMA.md` §4-6, `docs/architecture/db-specialist-review.md` §3

---

## Acceptance Criteria

- [ ] **AC1:** Firestore backup taken before any migration activity
  - Full export to Cloud Storage bucket
  - Verified: all country subcollections included
  - Backup timestamped and documented

- [ ] **AC2:** All Firestore collections exported to JSON
  - Countries with all nested subcollections (orders, products)
  - Users collection
  - Settings/config documents
  - Profit statistics document(s)
  - Expenses and expense categories

- [ ] **AC3:** Customer data deduplicated
  - Customers extracted from embedded order data
  - Deduplication by phone number (UNIQUE constraint)
  - Ambiguous matches flagged for manual review
  - Dedup report generated with merge decisions

- [ ] **AC4:** Data transformed to PostgreSQL format
  - Embedded objects → normalized rows
  - items[] arrays → order_items rows
  - Firestore Timestamps → PostgreSQL TIMESTAMPTZ
  - Firestore document IDs → PostgreSQL IDs (preserve original IDs where possible)
  - Country codes standardized

- [ ] **AC5:** Data loaded into Supabase in correct dependency order
  - Load order: countries → customers → products → orders → order_items → expenses → profit_stats
  - No FK violations during load
  - All constraints satisfied

- [ ] **AC6:** Data integrity verified
  - Row counts match: Firestore document count = PostgreSQL row count per table
  - Spot-check 10 random orders: all fields match
  - No orphaned order_items (all have valid order_id)
  - No orphaned orders (all have valid country_code, customer_id)
  - Checksums on critical numeric fields (totals, quantities)

- [ ] **AC7:** Supabase Realtime verified with migrated data
  - Subscribe to orders table → receive events on INSERT/UPDATE

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Subcollection data loss | Explicitly iterate all country documents, verify each has subcollections exported |
| Customer dedup errors | Manual review for ambiguous matches, preserve original data in audit field |
| Order ID conflicts | Check for duplicate Shopify IDs before migration |
| Data type mismatches | Transform script handles all known Firestore types → PostgreSQL |

## Testing

- Row count verification: every Firestore collection count = PostgreSQL table count
- Spot-check: 10 random orders verified field-by-field
- Constraint check: no FK violations, no NULL in required fields
- Dedup check: no duplicate phone numbers in customers table
- Realtime test: insert order via Supabase client → subscription receives event
