## Story

**As a** system administrator,
**I want** authentication migrated to Supabase with Row Level Security enforced at the database level,
**So that** every data access is authorized regardless of client-side code.

---

## Context

Firebase Auth uses custom claims for role management, stored in 3 places (Auth claims, users collection, roles_admin collection). Supabase consolidates this into a single `users.role` column enforced via RLS policies. The JWT-based `user_role()` helper function (Phase 5, Appendix D) enables performant policy checks without per-request database queries.

**Reference:** `docs/architecture/technical-debt-assessment.md` §3.3 Phase A, Appendix D

## Dev Notes

### Transition Architecture
During the transition period (Story 1.2 → Story 1.5):
- **Auth:** Supabase (login, middleware, Server Actions)
- **Data:** Firebase Firestore (client-side onSnapshot for real-time)
- **Bridge:** Firestore rules relaxed for reads (middleware protects routes). Writes still require Firebase Admin (Server Actions bypass rules).

### user_role() Performance
- JWT claim check: O(1) — no database query
- Fallback to users table: O(1) — indexed PK lookup
- `SECURITY DEFINER` allows function to read users table even with RLS enabled
- `STABLE` allows PostgreSQL to cache result within a statement

### JWT Role Injection
- Trigger on `auth.users` BEFORE UPDATE injects `user_role` and `assigned_countries` into `raw_app_meta_data`
- This means JWT claims are populated on session refresh, not immediately on user creation
- For new users: first request may use fallback path (users table query)

### RLS Policy Design
- Split settings policies (INSERT/UPDATE separate) per Phase 5 architecture correction
- FORNECEDOR country filtering uses subquery: `SELECT assigned_countries FROM users WHERE id = auth.uid()`
- Order items policy uses EXISTS subquery to check parent order access

### Admin User Creation
After deploying migrations, create admin user via Supabase Dashboard:
1. Go to Authentication > Users > Add user
2. Set email and password
3. Then insert into public.users: `INSERT INTO users (id, email, name, role) VALUES ('<auth-user-id>', 'email', 'name', 'ADMIN')`
