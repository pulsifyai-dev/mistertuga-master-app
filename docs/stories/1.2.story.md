## Story

**As a** system administrator,
**I want** authentication migrated to Supabase with Row Level Security enforced at the database level,
**So that** every data access is authorized regardless of client-side code.

---

## Context

Firebase Auth uses custom claims for role management, stored in 3 places (Auth claims, users collection, roles_admin collection). Supabase consolidates this into a single `users.role` column enforced via RLS policies. The JWT-based `user_role()` helper function (Phase 5, Appendix D) enables performant policy checks without per-request database queries.

**Reference:** `docs/architecture/technical-debt-assessment.md` §3.3 Phase A, Appendix D

---

## Acceptance Criteria

- [ ] **AC1:** Supabase Auth configured for email/password
  - Email templates customized (reset password, magic link if used)
  - Auth settings match business requirements

- [ ] **AC2:** `user_role()` helper function deployed
  - Reads role from JWT claim first (fast path)
  - Falls back to `users.role` column if JWT claim missing
  - `SECURITY DEFINER` + `STABLE` for performance
  - Test: Function returns correct role for each user type

- [ ] **AC3:** RLS policies active on ALL public tables
  - `orders`: ADMIN full access, FORNECEDOR read-only for assigned countries
  - `order_items`: Follows orders policy via FK
  - `customers`: ADMIN full access, FORNECEDOR read-only
  - `products`: ADMIN full access, FORNECEDOR read-only
  - `app_settings`: ADMIN read/write, FORNECEDOR read-only
  - `users`: ADMIN full access, users can read own profile
  - `profit_stats`: ADMIN only
  - `expenses` + `expense_categories`: ADMIN only
  - `countries`: All authenticated users can read
  - `audit_log`: ADMIN read-only (no direct writes — trigger-based)
  - Split settings: separate FOR INSERT and FOR UPDATE policies (Phase 5 correction)

- [ ] **AC4:** Next.js middleware authenticates all protected routes
  - `/(app)/*` routes require valid Supabase session
  - Login page accessible without auth
  - Middleware refreshes session tokens automatically
  - Unauthorized access redirects to login

- [ ] **AC5:** Admin self-registration vulnerability closed (TD-H6)
  - No environment variable code-based admin creation
  - Admin users created only via existing admin or database seed

- [ ] **AC6:** RLS tested with positive AND negative test cases
  - Each policy tested with test users per role (ADMIN, FORNECEDOR)
  - Verify denied operations return appropriate errors
  - Verify allowed operations succeed

## Testing

- Login as ADMIN → can CRUD orders, settings, users
- Login as FORNECEDOR → can read assigned country orders, cannot write
- Login as FORNECEDOR → cannot access profit_stats, expenses
- Attempt direct API call without auth → rejected
- Attempt to bypass RLS via Supabase client → denied
- Middleware redirects unauthenticated users to /login
